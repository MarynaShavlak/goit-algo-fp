# Завдання 6 — Жадібні алгоритми та динамічне програмування

Вибір страв із **максимальною сумарною калорійністю** в межах обмеженого
бюджету. Це класична задача **про рюкзак 0/1**: кожну страву можна взяти не
більше одного разу.

```python
items = {
    "pizza":     {"cost": 50, "calories": 300},
    "hamburger": {"cost": 40, "calories": 250},
    "hot-dog":   {"cost": 30, "calories": 200},
    "pepsi":     {"cost": 10, "calories": 100},
    "cola":      {"cost": 15, "calories": 220},
    "potato":    {"cost": 25, "calories": 350},
}
```

## Запуск

```bash
python task_6/main.py
```

## Підходи

**`greedy_algorithm(items, budget)`** — сортує страви за спаданням
співвідношення `калорії / вартість` і бере їх, поки вистачає бюджету. Якщо
страва не вміщується — **пропускає** її (`continue`), а не зупиняється, тож
може «добрати» дешевші страви далі по списку. Повертає список назв страв.

**`dynamic_programming(items, budget)`** — рюкзак 0/1. Зовнішній цикл по
стравах, внутрішній — по бюджету в **зворотному** порядку (від `budget` до
`cost`). Зворотний порядок гарантує, що кожну страву враховано щонайбільше один
раз. Повертає **гарантовано оптимальний** набір.

## Результат (бюджет = 100)

| Підхід | Набір | Вартість | Калорії |
|---|---|---:|---:|
| Жадібний | cola, potato, pepsi, hot-dog | 80 | **870** |
| ДП | pizza, pepsi, cola, potato | 100 | **970** |

Жадібний витрачає лише 80 з бюджету й набирає 870 ккал. Він рано бере дешеві
страви з найкращим співвідношенням (`cola`, `potato`, `pepsi`, `hot-dog`) і вже
не може докупити нічого дорожчого — на залишок 20 не вміщується ні
`hamburger` (40), ні `pizza` (50). Динамічне програмування використовує весь
бюджет і знаходить кращий набір на **970 ккал** — на 100 ккал більше. Це
наочно показує, що **жадібний підхід не гарантує оптимуму** для задачі 0/1
knapsack.

## Аналіз рішення

### Сильні сторони

* Обидві функції коректні. У ДП правильно обрано **зворотний** внутрішній цикл —
  саме він забезпечує обмеження «кожна страва не більше разу» (рюкзак 0/1, а не
  unbounded).
* У жадібному вдало використано `continue` замість `break`: алгоритм не
  зупиняється на першій завеликій страві, а продовжує перебір — на практиці це
  пакує більше страв.

### Що покращено / варто враховувати

1. **Валідація бюджету.** Для `budget < 0` вираз `[0] * (budget + 1)` давав би
   порожній список і подальший `IndexError`. Додано перевірку → `ValueError`.
2. **Ділення на нуль у жадібному.** Якщо у страви `cost == 0`, обчислення
   `calories / cost` впаде. У ключі сортування додано захист: безкоштовні страви
   отримують найвищий пріоритет (`float("inf")`).
3. **Пам'ять у ДП.** Рядок `chosen[b - cost] + [name]` створює **новий список**
   на кожному покращенні, тож одночасно зберігається `budget + 1` списків →
   **O(n · budget)** пам'яті. Для невеликих бюджетів це нормально. Для дуже
   великих краще тримати лише числову таблицю `dp` і **відновлювати** набір у
   кінці зворотним проходом — тоді пам'ять для шляху не залежить від кількості
   страв.
4. **Зручний вивід.** Додано хелпер `summarize`, що рахує сумарну вартість і
   калорійність — щоб одразу бачити, який набір вигідніший.

## Складність (Big-O)

| Алгоритм | Час | Пам'ять | Оптимальність |
|---|---|---|---|
| `greedy_algorithm` | O(n log n) | O(n) | не гарантована |
| `dynamic_programming` | O(n · budget) | O(budget) | гарантований оптимум |

де `n` — кількість страв.

## Висновок

Жадібний алгоритм швидкий (`O(n log n)`, не залежить від бюджету), але дає лише
наближення й тут програє 100 ккал. Динамічне програмування повільніше
(`O(n · budget)` — псевдополіноміальна складність, що зростає з бюджетом), проте
знаходить **точний максимум**. Для задачі вибору їжі з фіксованим набором
страв і помірним бюджетом правильний вибір — **динамічне програмування**;
жадібний доречний, коли потрібна миттєва відповідь і допустима похибка.
