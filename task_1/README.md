# Завдання 1 — Структури даних. Сортування. Однозв'язний список

Для однозв'язного списку реалізовано три операції:

1. **`reverse`** — реверсування списку зміною посилань між вузлами;
2. **`insertion_sort`** — сортування вставками;
3. **`merge_sorted_ll`** — об'єднання двох відсортованих списків в один.

## Запуск

```bash
python task_1/main.py
```

Залежностей немає — лише стандартна бібліотека Python.

## Як це працює

**`reverse`** іде списком і для кожного вузла розвертає його посилання `next` на
попередній (`prev → current → next_node`). Наприкінці новим `head` стає колишній
хвіст. O(n) часу, O(1) пам'яті, без створення чи видалення вузлів.

**`insertion_sort`** будує новий відсортований ланцюг, **перез'єднуючи існуючі
вузли**: кожен вузол вставляється на потрібне місце у вже відсортовану частину.
O(n²) часу, O(1) додаткової пам'яті.

**`merge_sorted_ll`** іде двома списками одночасно й щоразу бере менше з двох
поточних значень, додаючи його в кінець результату через вказівник на хвіст
(`tail`) — тож вставка коштує O(1), а все злиття — O(n + m).

## Результат

```
Зв'язний список:           5 -> 15 -> 10 -> 35 -> 25
Обернений список:          25 -> 35 -> 10 -> 15 -> 5
Відсортований (вставками):  5 -> 10 -> 15 -> 25 -> 35
Злиття двох відсортованих:  5 -> 5 -> 10 -> 12 -> 13 -> 14 -> 14 -> 15 -> 23 -> 25 -> 35 -> 35 -> 37 -> 39 -> 41
```

## Аналіз рішення

### Сильні сторони

* `merge_sorted_ll` логічно коректний: правильно порівнює голови двох списків і
  дозбирає залишок.
* Реалізовано повний набір базових операцій списку (вставка, видалення, пошук).
* Реверс за результатом давав правильний обернений список.

### Що покращено

1. **Реверс тепер справді змінює посилання.** Завдання прямо вимагає реверсувати
   «змінюючи посилання між вузлами», а вихідна версія натомість у циклі
   **видаляла вузли за значенням і вставляла їх заново** після хвоста. Результат
   був правильний, але:
   * це не відповідає умові (посилання напряму не змінювались);
   * складність — **O(n²)** (кожен `delete_node` проходить список).

   Новий `reverse` — класичне перез'єднання вказівників за **O(n)** і **O(1)**
   пам'яті.

2. **Виправлено баг сортування на однакових значеннях.** Вихідний
   `insertion_sort` (як і реверс) спирався на `delete_node(current.data)`, який
   видаляє **перший** вузол із таким значенням — не обов'язково `current`. За
   наявності дублікатів це псувало список. Наприклад:

   | Вхід | Вихідний код | Правильно |
   |---|---|---|
   | `[1, 2, 1]` | `[1, 2, 1]` ❌ | `[1, 1, 2]` |
   | `[3, 1, 3, 1, 2]` | `[1, 2, 3, 3, 1]` ❌ | `[1, 1, 2, 3, 3]` |
   | `[5, 5, 2, 8, 2]` | `[2, 5, 5, 8, 2]` ❌ | `[2, 2, 5, 5, 8]` |

   (У вихідному демо баг не спливав, бо `insertion_sort` викликався лише на
   списку без дублікатів.) Новий варіант перез'єднує **самі вузли**, тож працює
   коректно й з дублікатами — перевірено стрес-тестом на 2000 випадкових списків.

3. **Злиття стало O(n + m) замість O(n²).** Вихідна версія додавала елементи
   через `insert_at_end`, який щоразу проходить увесь зібраний список (O(n²) на
   все злиття). Додано вказівник на хвіст — вставка за O(1).

4. **Дрібниці.** Прибрано зайві присвоєння `cur = None` (вони нічого не робили);
   додано `to_list()` і компактний `print_list` (вивід «5 -> 10 -> 15» одним
   рядком замість друку кожного елемента окремо).

### Зауваження

* `merge_sorted_ll` не змінює вхідні списки (копіює значення). Якщо потрібна
  економія пам'яті — можна **перез'єднати** наявні вузли замість копіювання
  (O(1) додаткової пам'яті), але тоді вихідні списки «розберуться».
* Для дуже великих списків замість сортування вставками (O(n²)) варто взяти
  **сортування злиттям**, яке для зв'язних списків працює за O(n log n).

## Складність (Big-O)

| Операція | Час | Пам'ять |
|---|---|---|
| `reverse` | O(n) | O(1) |
| `insertion_sort` | O(n²) | O(1) |
| `merge_sorted_ll` | O(n + m) | O(n + m) |

де `n`, `m` — довжини списків.
